# 设计说明

## 1. 项目定位

在 **完全独立仓库** 中实现《教育知识库项目实战需求说明》中的能力：多类型内容导入（含任务与元数据）、课程/文档/题库检索、基于库内内容的问答与引用、多轮与流式输出。

一期数据源以 **Word（.docx）与 Markdown** 为主；**PDF 仅保留配置与接口层面的扩展点**，不在首版实现解析逻辑。

## 2. 与 `knowledge` 的关系

- **仓库**：独立，不 submodule、不共享包（除非日后你主动抽公共库）。
- **可借鉴**：FastAPI 双服务或路由拆分、LangGraph 导入/查询流水线、Milvus 混合检索与重排序、任务状态与 SSE 等模式。
- **差异**：教育领域元数据（课程名、项目名、章节、题库名、题目编码、题型等）与「商品名」类场景不同；题库与课程目录宜 **结构化存储 + 向量检索** 配合，而非单一 chunk 混排。

## 3. 目录规划（意图）

| 路径 | 用途（规划） |
|------|----------------|
| `api/` | HTTP 入口：导入、查询、健康检查等 |
| `processor/` | LangGraph：导入图、查询图（后续） |
| `schema/` | Pydantic 请求/响应与领域模型 |
| `services/` | 业务编排：上传、解析调度、备份等 |
| `utils/` | 客户端单例、任务 ID、SSE、Milvus 封装等 |
| `prompts/` | 提示词模板 |
| `temp_data/` | 上传与中间产物（运行时，不入库敏感数据） |
| `docs/` | 设计与配置文档（本目录） |

## 4. 数据源策略

- **Markdown**：正文解析 → 章节/标题感知切分 → 元数据绑定 → 向量与（如需要）结构化字段。
- **Word**：.docx → 抽取为可分段文本（保留层级）→ 与 MD 共用后续管线。
- **PDF**：不实现解析；在 API/适配器层保留「可接入」形状（例如根据扩展名路由到 PDF 适配器，在未实现时返回明确未实现语义）。环境变量 **`PDF_IMPORT_ENABLED`** 为预留开关，**默认关闭**，见 [CONFIG.md](CONFIG.md)。

## 5. 存储与检索（思路）

- **向量库（如 Milvus）**：文档片段、讲义等语义检索；过滤条件绑定课程/章节等元数据。
- **结构化库（如 MongoDB 或关系库）**：课程目录、章节树、题库与题目字段（题型、选项、答案、解析、题目编码等）；列表页、详情页、按题型筛选优先走结构化查询。
- **问答**：LLM 生成时强制引用格式（课程名、章节、文件名、题号等），与需求中的可追溯一致。

## 6. 导入任务与容错

- 异步任务 + 可轮询状态 ID；单任务失败不应导致整进程不可用。
- PDF 路径：开关关闭或适配器未实现时，应 **快速失败并返回明确原因**，避免静默挂起。

## 7. 后续扩展（不在首版范围）

后台管理、增量更新、权限与多租户、视频字幕检索、多模态等，可在 Schema 与 API 上预留扩展点，避免推翻重来。

## 8. 当前状态

仅 **目录占位 + 文档**；`.env.example` 已包含 `PDF_IMPORT_ENABLED`（预留，默认关闭）。应用代码待评审本设计后再编写。
